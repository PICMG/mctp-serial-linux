# libsermctp

libsermctp is a small C++ library that implements MCTP serial framing, reassembly, and helpers to bridge between a serial transport and the Linux kernel MCTP subsystem. It is used by the `mctp-bridge` application and by the example programs in this repository.

## Purpose

- Provide a reusable, public C++ API for encoding/decoding MCTP frames on a serial link.
- Offer components for building user-space bridges and test fixtures without exposing internal helper types.

## Project layout

- `include/sermctp/` — public headers for the library API (MctpBridge, MctpFramer, LinuxMctpSerial wrappers, etc.).
- `include/sermctp/detail/` — internal headers; these are intentionally not required by consumers.
- `src/` — library implementation sources.
- `examples/` — small example programs demonstrating usage (e.g. `linux_to_tty_example.cpp`, `tty_to_linux_example.cpp`, broadcast examples).
- `tests/` — unit and integration tests used during development.

## Building

Prerequisites:

- CMake >= 3.16
- A C++17-capable compiler (g++/clang)
- `pkg-config`
- `libnl` development packages (if building features that talk to netlink)

Out-of-tree build

Note: the library's top-level CMakeLists.txt is located in `libsermctp/src/`, not in the `libsermctp/` directory. Choose one of the following equivalent ways to configure and build.

From the repository root (single-line invocation):

```bash
cmake -S linux-mctp/libsermctp/src -B linux-mctp/libsermctp/build -DCMAKE_BUILD_TYPE=RelWithDebInfo
cmake --build linux-mctp/libsermctp/build -- -j$(nproc)
```

Or change into the `libsermctp` directory and configure from `src/`:

```bash
cd linux-mctp/libsermctp
mkdir -p build
cd build
cmake -S ../src -B . -DCMAKE_BUILD_TYPE=RelWithDebInfo
cmake --build . -- -j$(nproc)
```

This produces either a shared `libsermctp.so` (default) and/or a static `libsermctp.a` depending on CMake options.

### Shared vs static

- Default: the build creates a shared library target named `sermctp` (`add_library(sermctp SHARED ...)`). That produces the runtime/shared object that most consumers will link to.
- Optional static archive: enable `BUILD_STATIC_LIBS` to also produce a static archive target named `sermctp_static`:

```bash
cmake -S linux-mctp/libsermctp/src -B linux-mctp/libsermctp/build -DBUILD_STATIC_LIBS=ON
cmake --build linux-mctp/libsermctp/build -- -j$(nproc)
```

When `BUILD_STATIC_LIBS=ON` the build will produce a static archive (typically named `libsermctp_static.a`) alongside the shared library in the build tree. If you only need the static archive (and not the shared object), there is no dedicated project option to disable the shared target — the CMakeLists currently always creates the shared target — so building static-only requires editing the CMake files or extracting the static archive from the build outputs.

Locating artifacts:

```bash
cd linux-mctp/libsermctp/build
# find the shared object
find . -name 'libsermctp*.so' -print
# find the static archive
find . -name 'libsermctp*.a' -print
```

Linking notes:

- If you install the library (via `cmake --install .` or your distribution packaging), the `pkg-config` file generated by the project will point consumers at the correct library and flags. When linking directly against the static archive, ensure you pull in any required system libraries (threading, rt, etc.) as static linking can require extra linker flags.

## Examples

Examples live under `libsermctp/examples/`. They show how to:

- Create a framer bound to a PTY and exchange MCTP frames.
- Run a bridge-like program that forwards between a PTY and the kernel-facing interface.
- Send and receive broadcast messages via the bridge's UNIX datagram interface.

See individual example files' headers for usage notes and command-line flags.

## Tests

The test suite lives in `libsermctp/tests/` and can be enabled via CMake. The project configures tests from the `src/` directory, so be sure to point `cmake` at `libsermctp/src` when configuring a build. These tests require root to run (they exercise kernel/serial interactions), so run them under `sudo`.

From the repository root (recommended):

```bash
# create an out-of-tree build dir and configure (points at src/ where CMakeLists.txt lives)
cmake -S linux-mctp/libsermctp/src -B linux-mctp/libsermctp/build -DCMAKE_BUILD_TYPE=RelWithDebInfo -DBUILD_TESTS=ON

# build the library + tests
cmake --build linux-mctp/libsermctp/build -- -j$(nproc)

# run the tests with sudo (recommended; required for some integration tests)
sudo ctest --test-dir linux-mctp/libsermctp/build --output-on-failure
```

Or, from the `libsermctp` directory:

```bash
cd linux-mctp/libsermctp
mkdir -p build
cd build
cmake -S ../src -B . -DCMAKE_BUILD_TYPE=RelWithDebInfo -DBUILD_TESTS=ON
cmake --build . -- -j$(nproc)
sudo ctest --output-on-failure
```

Run the aggregated test binary directly (if you prefer): locate `test_runner` in the build tree and run it with `sudo`:

```bash
find linux-mctp/libsermctp/build -type f -executable -name test_runner -print
sudo linux-mctp/libsermctp/build/path/to/test_runner
```

Notes:

- Tests exercise kernel and serial interactions and therefore require root privileges; run them with `sudo`.
- If you need to list tests without running them, use `ctest -N` (you can do this without sudo to inspect which tests are present).
- If you want to skip root-only tests, they are labeled `requires_root` in the test registration; you can exclude them by running `ctest -LE requires_root --output-on-failure` (note: this will skip tests that need root).
- The CMake option used by this project is `BUILD_TESTS`; enabling it (as shown above) arranges for the `test_runner` target to be built and registered with CTest.

## Notes

- The public API is intentionally small. Internal helpers are kept in `detail/` and are not required by typical consumers.
- If you need to interact with low-level kernel MCTP constructs (AF_MCTP sockets), consult the `mctp-bridge/examples/python_mctp_bridge_example.py` script which demonstrates a libc-based AF_MCTP usage pattern.
